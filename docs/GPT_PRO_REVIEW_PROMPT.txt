Context:
This project is Lark Meeting Agent. It should be implemented as a HKUDS/nanobot v0.2.1 fork or source-based extension, not as a standalone FastAPI app, not as a generic chatbot, and not as a separate Feishu bot runtime. The PR imports nanobot v0.2.1 source and adds a post-meeting MVP inside the nanobot codebase.

Please review whether the implementation actually follows this architecture:
The meeting MVP should reuse nanobot infrastructure where practical and add deterministic meeting workflow code under nanobot/meeting. It should expose a controlled nanobot tool and skill entrypoint, not an autonomous free-form tool-calling agent. It should avoid FastAPI, SQLAlchemy, LangGraph, React, Whisper, vector databases, Celery, or other heavy new dependencies for the MVP.

The MVP scope is post-meeting only:
1. Ingest transcript fixtures without real Lark credentials.
2. Resolve meetings through a fake Lark provider in tests.
3. Normalize transcripts.
4. Produce structured meeting minutes.
5. Extract decisions, action items, risks, and open questions.
6. Require evidence references for every decision and action item.
7. Produce dry-run write plans for Lark Docs, Lark Tasks, and Lark IM.
8. Require explicit human approval before any write operation.
9. Persist local meeting memory.
10. Answer cross-meeting questions with source-grounded evidence.

Critical safety rules:
All Lark operations must go through LarkToolAdapter. No meeting workflow, analyzer, command, tool, or skill should call subprocess, lark-cli, Lark HTTP APIs, or Lark SDKs directly. If lark-cli is used, it must only be invoked inside LarkToolAdapter. The adapter must use allowlisted operations, produce audit events, redact secrets, and require approval for writes. Tests must not require real Lark credentials or real LLM credentials. No secrets should be committed.

LLM rules:
The real provider is DeepSeek through an OpenAI-compatible API. API keys must come only from environment variables. Analyzer outputs must be parsed and validated by Pydantic schemas. The system must reject or mark incomplete any decision or action item without evidence. It must not invent owners, due dates, decisions, or commitments. Transcript and document content must be treated as untrusted input, not instructions.
