# Design: Implement OAPI Lark Provider

## Current State

`LarkToolAdapter` already enforces:

- operation allowlists,
- read/write classification,
- write approval,
- dry-run status,
- audit events,
- payload redaction.

Provider implementations currently include `FakeLarkProvider` and `CliLarkProvider`.

## Design

Add `OapiLarkProvider` as another provider implementation.

```text
Workflow / tool
  -> LarkToolAdapter
      -> FakeLarkProvider
      -> CliLarkProvider
      -> OapiLarkProvider
           -> OapiHttpRunner
```

The provider maps adapter operation names to Lark OpenAPI request specs:

- `auth.status`
- `calendar.agenda`
- `vc.search`
- `vc.notes`
- `minutes.search`
- `docs.search`
- `docs.fetch`
- `docs.create`
- `task.search` / `task.list`
- `task.create`
- `im.send`

The first implementation is intentionally thin:

- standard-library `urllib.request`,
- bearer token injected from constructor or `LARK_OAPI_ACCESS_TOKEN`,
- configurable base URL for tests,
- bounded timeout,
- fake HTTP runner in tests,
- no token refresh logic.

## Safety

- `dry_run=True` for write operations returns request preview and does not call HTTP.
- `dry_run=False` still requires adapter approval.
- Unknown operations are rejected by adapter/provider allowlists.
- Audit logs are generated by `LarkToolAdapter` and redacted before persistence.

## Trade-offs

- Endpoint payloads are MVP mappings, not full SDK coverage.
- Token lifecycle is deliberately outside this change.
- Real smoke remains opt-in because CI must not require Lark credentials.
